# Project Context
This project is a line movement worker for sports betting. It consumes updates from the `odds_prop_history` table in Postgres, tracking every change to a prop's line or price. The worker summarizes each change along with the historical context for that prop, enabling the identification of signals and patterns in line or price movement. The system is designed for modularity, testability, and robust data handling, supporting advanced analysis of betting market dynamics.

# Line Movement Worker Coding Instructions

## General Principles
- All code must follow SOLID principles.
- Use dependency injection for all external dependencies (e.g., database connections, services).
- Practice Test Driven Development (TDD):
  - Write tests first using pytest and pytest-mock.
  - Ensure all new code is covered by tests.

## Database and Data Handling
- Use SQLAlchemy ORM for all database table interactions (including odds_prop_history).
- Always use Pydantic schemas for data validation and serialization when passing data between code layers (e.g., from DB to business logic, or to API).

## Example Prompts
- "Generate a new worker module that follows SOLID and uses dependency injection."
- "Add a new feature, but write the pytest test first."
- "Refactor this DB access to use SQLAlchemy and Pydantic."

## Project Structure
- `schemas/`: All Pydantic schemas for data validation and serialization.
- `db/`: Database engine configuration and connection logic.
- `db/models/`: All SQLAlchemy ORM models.
- `workers/`: Consumer logic for processing prop updates and changes.
- `analyzers/`: Business logic for summarizing and analyzing line/price movement.
- `tests/`: All unit tests (pytest, pytest-mock) for TDD.
- `interfaces/`: All interface classes for accessing and abstracting data sources (e.g., database access interfaces)
